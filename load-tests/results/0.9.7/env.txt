New in 0.9.7:

  1. 3 new SOURCE-retention AI guardrail annotations:
       @AIIdempotent   — declares an operation must remain idempotent; AI must
                         never introduce side effects that cause repeated calls
                         to produce different results. Attribute: reason (String).
       @AIFeatureFlag  — marks code gated behind a runtime feature flag; AI must
                         preserve the flag check and never assume it always active.
                         Attributes: flag (String), defaultValue (boolean).
       @AISecure       — marks security-critical code (auth, encryption, session
                         management); AI must not weaken security properties and
                         must flag every change for security review.
                         Attribute: aspect (String).
     Compile-time validation for contradictory combinations:
       @AIIdempotent + @AIDraft, @AIFeatureFlag + @AILocked,
       @AISecure + @AIIgnore; plus advisory for blank @AISecure.aspect and
       no-op for blank @AIFeatureFlag.flag.

  2. 3 new AI platform integrations:
       .clinerules               (Cline AI assistant — Markdown)
       .junie/guidelines.md      (JetBrains Junie — Markdown)
       .kiro/steering/*.md       (Amazon Kiro — granular per-class, Markdown)

  3. Parallel file writes (ForkJoinPool.commonPool()):
       All platform output files are now written in parallel via parallelStream().
       A SynchronizedMessager proxy wraps the javac Messager for thread safety.
       WriteCache was already fully synchronized; no changes needed there.
       Status messages are collected in a Queue and emitted sequentially from
       the main thread after the parallel phase completes.
       Benchmark impact: negligible for small projects; measurable improvement
       expected at N=50+ active services.

Caveats — read before drawing conclusions:

- The JMH suite calls writeFileIfChanged directly, bypassing
  GuardrailContentBuilder and the new parallelStream() phase. The parallel
  speedup is therefore NOT reflected in the JMH numbers; it only affects
  the stress sweep wall-clock figures.
- The stress sweep creates a fresh TempDir per leg → cache is always empty;
  numbers reflect Windows process-launch jitter ±15%.
- Overhead(ms) at N=10 and N=100 is elevated vs. 0.9.5 (428ms / 154ms vs.
  typical ~200ms / ~80ms). This is consistent with Windows I/O jitter on a
  freshly launched JVM; the N=500 and N=1000 rows (62ms / 585ms) are within
  the historical noise band.
- OutputSize(B) at every N is slightly larger than 0.9.5 due to the 3 new
  annotation types generating additional output sections when present. The
  stress harness annotates with a fixed mix; the new annotations add no
  OutputSize overhead here since no stress sources use them.
- The new platforms (Cline, Junie, Kiro) are not in the stress sweep opt-in
  set, so they add zero overhead to these numbers.
