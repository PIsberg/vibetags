# Load-test baselines

One folder per processor release. Each folder is a **frozen baseline** captured by running the harness in `load-tests/` against the matching `vibetags-processor` artifact.

## Layout

```
results/
├── README.md                       ← you are here
├── 0.5.4/  0.5.5/  0.5.6/         (early baselines — pre-BOM split)
│   ├── env.txt                     machine + JDK + commit metadata
│   ├── stress.txt                  AnnotationVolumeStressTest table (N=10..1000)
│   ├── memory.txt                  MemoryVolumeStressTest table (N=10..1000)
│   ├── concurrent.txt / .xml       ConcurrentBuildTest surefire
│   └── jmh.json                    ProcessorHotPathBenchmark JMH JSON
├── 0.7.0/                          (BOM split + @AIContract + 10 new platforms)
│   └── (same files)
├── 0.7.1/                          (write cache + streaming compare + pre-sized buffers)
│   ├── (same files plus)
│   ├── jmh-cache-hit.json          WriteCacheHitBenchmark JMH JSON (small/medium/large × marker/non-marker)
│   └── jmh-cache-hit-summary.md    Human-readable analysis of the cache benchmark
└── _plots/                         regenerated by tools/plot-{results,cache-hit}.py
    ├── overhead-vs-n.png           (1) Stress overhead vs. N, all releases
    ├── hotpath-by-release.png      (2) JMH hot-path bar chart, all releases (log-y)
    ├── writeFileIfChanged-detail.png (3) writeFileIfChanged variants only (linear-y)
    ├── memory-overhead-vs-n.png    (4) Allocation overhead vs. N, all releases
    ├── memory-peak-heap-vs-n.png   (5) Peak heap vs. N, all releases
    ├── cache-hit-time.png          (6) Cache-hit wall-clock proof (since 0.7.1)
    └── cache-hit-alloc.png         (7) Cache-hit allocation proof (since 0.7.1)
```

**No `0.6.0` baseline:** the 0.6.0 release was a packaging change (split into BOM + annotations + processor) with no processor-source changes, so capturing a baseline would have been redundant with `0.5.6`.

## How these were captured

```bash
TAG=0.7.2

# 1. Install the target processor version into the local Maven repo
#    (or use git worktree if measuring an older tagged release)
cd vibetags-annotations && mvn install -DskipTests
cd ../vibetags             && mvn install -DskipTests

# 2. Run stress + memory + concurrent capped at N=1000
cd ../load-tests
mvn test -Dprocessor.version=$TAG \
         -Dtest=AnnotationVolumeStressTest,MemoryVolumeStressTest,ConcurrentBuildTest \
         -Dstress.max.classes=1000

# 3. Run JMH benchmarks with GC profiler (writes JSON directly into results/$TAG/)
mvn package -DskipTests -Dprocessor.version=$TAG
java -jar target/benchmarks.jar -wi 3 -i 5 -f 1 -tu us -bm avgt -prof gc \
     -rf json -rff results/$TAG/jmh.json
java -jar target/benchmarks.jar WriteCacheHitBenchmark -wi 3 -i 5 -f 1 \
     -tu us -bm avgt -prof gc \
     -rf json -rff results/$TAG/jmh-cache-hit.json

# 4. Copy stress/memory/concurrent artefacts (JMH JSONs already in place)
cp $(ls -t target/stress-results-*.txt | head -1) results/$TAG/stress.txt
cp $(ls -t target/memory-results-*.txt | head -1) results/$TAG/memory.txt
cp target/surefire-reports/se.deversity.vibetags.loadtest.ConcurrentBuildTest.txt      results/$TAG/concurrent.txt
cp target/surefire-reports/TEST-se.deversity.vibetags.loadtest.ConcurrentBuildTest.xml results/$TAG/concurrent.xml

# 5. Hand-write env.txt with java -version + uname/CPU + commit sha + brief notes
```

## Comparison across all released baselines

Same machine (i7-1260P), JDK Temurin 26, cap `stress.max.classes=1000`.

`OutputSize(B)` is byte-identical across every release at every N
(17 156 / 122 555 / 599 895 / 1 196 918), so the work product is unchanged — only
the cost varies.

### Stress sweep — `Overhead(ms)` (processor − baseline)

| N | 0.5.4 | 0.5.5 | 0.5.6 | 0.7.0 | 0.7.1 |
|---:|---:|---:|---:|---:|---:|
| 10 | 730 | 750 | 432 | 425 | 445 |
| 100 | -38 | -184 | 183 | 228 | 217 |
| 500 | 1062 | 1859 | 283 | 16 | 376 |
| 1000 | 933 | 1209 | 211 | 13 | 333 |

The 0.7.0 N=500/1000 numbers were anomalously low compared to historical and 0.7.1
runs — a one-off measurement on an unusually quiet machine. The shape of the curve
(flatter at high N) is consistent across 0.5.6 onwards; absolute deltas should be
read with the documented Windows process-launch jitter (±15%).

## Memory comparison across all baselines

Same machine / JDK / window as the wall-clock numbers above. Cap: `stress.max.classes=1000`. Captured by `MemoryVolumeStressTest`.

### Allocation overhead — `OverheadAlloc(KB)` (processorAlloc − baselineAlloc)

| N | 0.5.4 | 0.5.5 | 0.5.6 | 0.7.0 | 0.7.1 |
|---:|---:|---:|---:|---:|---:|
| 10 | 811 | 837 | 790 | 2524 | 3275 |
| 100 | 21 394 | 21 584 | 20 825 | 19 342 | 24 278 |
| 500 | 96 188 | 96 356 | 93 351 | 93 381 | 99 141 |
| 1000 | 192 424 | 192 478 | 185 676 | 185 800 | 192 707 |

Allocation overhead scales linearly in N across all releases (see
`memory-overhead-vs-n.png`). 0.5.6 shaved ~3% off the per-class allocation of
0.5.4/0.5.5; 0.7.0/0.7.1 land within ±5% of 0.5.6 — the cache writes a small
sidecar entry (~80 bytes per platform per build) but doesn't move this metric
materially. The N=10 numbers tick up because cache initialisation amortises poorly
on tiny workloads; at N≥100 the slope converges.

### Peak heap during processor run — `PeakHeap(MB)`

| N | 0.5.4 | 0.5.5 | 0.5.6 | 0.7.0 | 0.7.1 |
|---:|---:|---:|---:|---:|---:|
| 10 | 20 | 19 | 19 | 21 | 21 |
| 100 | 33 | 37 | 37 | 31 | 31 |
| 500 | 149 | 82 | 79 | 147 | 132 |
| 1000 | 189 | 185 | 189 | 192 | 186 |

Peak-heap is GC-timing dependent — the 0.5.4/0.7.0 spikes at N=500 are GC
artefacts, and the five releases reconverge at N=1000. Use the allocation-overhead
chart for regression decisions; treat peak-heap as a sanity check for OOM risk only.

## Cache-hit benchmark (since 0.7.1)

`WriteCacheHitBenchmark` measures `writeFileIfChanged` with a real `WriteCache`
wired through (`writerWithCache`) vs. `null` (`writerNoCache`), at three body
sizes × two file types. 100 calls per JMH `@Benchmark` invocation
(`@OperationsPerInvocation(100)`) to amortise the framework's per-call ~150 µs
floor. Captured in `0.7.1/jmh-cache-hit.json`; analysis in
`0.7.1/jmh-cache-hit-summary.md`.

### Wall-clock (cache hit vs. no cache, lower is better)

| Body | File type | cache hit | no cache | speedup |
|---|---|---:|---:|---:|
| 1 KB | `.md` (marker) | 16.4 µs | 208.5 µs | **13×** |
| 1 KB | `.cursorrules` (non-marker) | 10.0 µs | 209.4 µs | **21×** |
| 12 KB | `.md` | 18.1 µs | 262.7 µs | **15×** |
| 12 KB | `.cursorrules` | 17.4 µs | 297.3 µs | **17×** |
| 1 MB | `.md` | 18.6 µs | 3 405 µs | **183×** |
| 1 MB | `.cursorrules` | 18.1 µs | 3 285 µs | **181×** |

### Allocation (lower is better)

| Body | File type | cache hit | no cache | reduction |
|---|---|---:|---:|---:|
| 1 KB | `.md` | 752 B/op | 11 449 B/op | **15×** |
| 12 KB | `.md` | 752 B/op | 101 553 B/op | **135×** |
| 1 MB | `.md` | 752 B/op | 8 392 001 B/op | **11 159×** |

The cache-hit path is constant time (~16-19 µs) and constant allocation (~750
B/op) regardless of body size — bounded by one `Files.readAttributes` syscall plus
an O(1) cached `String.hashCode()` lookup. The no-cache path scales linearly with
body size because it must `readString` the entire file every call.

## Caveats — read before drawing conclusions

1. **JMH error bars vary wildly across releases** because each baseline was measured during a different machine-state window. The 0.5.5 `writeFileIfChanged_smallWrite` ±5186 µs/op is system noise, not a real 5× regression. The narrow-error metrics across all releases (`buildServiceFileMap`, the 0.7.1 `_noChange` and `_smallWrite`) are trustworthy. When in doubt, re-run on a quiet machine.
2. **Stress times include process-launch jitter on Windows** — a single high-overhead leg (e.g. 0.7.0 N=500=16ms vs 0.7.1 N=500=376ms) is not enough to call a regression on its own. CI should accept ±15% variance, and large deltas should be reproduced before fixing.
3. **JUnit `@TempDir` cleanup fails on Windows** because `vibetags.log` is held open by the file logger during processing. `concurrent.xml` reports `errors=1` for this reason; the test body itself completes and assertions pass. Same applies to the per-N stress test legs in `surefire-reports/` (not stored here). Pre-existing harness issue affecting Windows runs only.
4. **All baselines used JDK Temurin 26**. Re-baseline on the JDK that CI uses (currently 21/25/26 across the matrix) before comparing CI artifacts against these numbers.
5. **No 0.6.0 baseline exists** — that release was a packaging change with no processor-source changes, so its perf profile is identical to 0.5.6.

## Diffing two baselines

For stress / memory: the plots overlay every committed release at once (run
`tools/plot-results.py`). Eyeball-comparing `stress.txt` files works for N ≤ 6
rows. For JMH, the canonical diff is the official `jmh-compare` (or
[JMH Visualizer](https://jmh.morethan.io/)) which understands confidence intervals
— paste both `jmh.json` files into the visualizer and look for benchmarks where
the bars don't overlap.

## Plots

Seven PNGs are generated in `_plots/` from the data in this folder:

| Image | Source data | What it shows |
|---|---|---|
| `overhead-vs-n.png` | `*/stress.txt` | Overhead (ms) vs. N (log x), one line per release. Reveals scaling shape. |
| `hotpath-by-release.png` | `*/jmh.json` | All six `ProcessorHotPathBenchmark` benchmarks, log-y bar chart with error bars, grouped by release. The shared scale makes large/small benchmarks comparable. |
| `writeFileIfChanged-detail.png` | `*/jmh.json` | The three `writeFileIfChanged` variants on a linear y-axis. Used to inspect the variants in detail and read off error bar magnitude directly. |
| `memory-overhead-vs-n.png` | `*/memory.txt` | Allocation overhead (MB, processor − baseline) vs. N (log x). The deterministic memory metric — flat curves between releases means no allocation regression. |
| `memory-peak-heap-vs-n.png` | `*/memory.txt` | Peak heap used during the processor compile (MB) vs. N. Noisier than allocation overhead because GC timing varies, but it is the number that decides whether a multi-module build OOMs. |
| `cache-hit-time.png` _(since 0.7.1)_ | `0.7.1/jmh-cache-hit.json` | Cache-hit vs. no-cache wall-clock per `writeFileIfChanged` call, log-y, with speedup ratios annotated above each pair. Six scenarios: (small / medium / large) × (marker / non-marker). |
| `cache-hit-alloc.png` _(since 0.7.1)_ | `0.7.1/jmh-cache-hit.json` | Same six scenarios on the `gc.alloc.rate.norm` axis. Shows the 10 000× allocation reduction at 1 MB body size. |

**Read the error bars before drawing conclusions.** Multiple historical bars
overlap once whiskers are drawn (e.g. 0.5.5 `writeFileIfChanged_smallWrite` vs.
0.5.4) — the apparent regression is inside measurement noise. This is exactly why
the chart is the right artefact, not the table.

### Regenerating

```bash
python -m pip install matplotlib   # one-time

# Plots 1-5 (release-trend, reads every results/<x.y.z>/ folder)
python tools/plot-results.py

# Plots 6-7 (cache-hit proof, reads results/0.7.1/jmh-cache-hit.json)
python tools/plot-cache-hit.py
```

`plot-results.py` auto-discovers any `<x.y.z>/` subdirectory containing
`stress.txt` and `jmh.json`, so adding a new release baseline later means
dropping a new folder and re-running — no script edits needed.
`plot-cache-hit.py` reads the latest available `jmh-cache-hit.json` (currently
pinned to `0.7.1/`); update the path inside the script when bumping baselines.
